I'm currently on [medical leave recovering from surgery](/blog/2026/killing-my-inner-necron/). Before I went under, I wanted to ship one thing I'd been failing to build for months: a sponsor panel at [sponsors.xeiaso.net](https://sponsors.xeiaso.net). Three previous attempts had drowned in GraphQL pain. This time I vibe coded it — pointed agent teams at the problem with prepared skills and let them generate the gnarly code I couldn't write myself.

And it works.

## The GraphQL swamp

Go and GraphQL are oil and water. I've held this opinion for years and nothing has changed it. The library ecosystem is a mess: [shurcooL/graphql](https://github.com/shurcooL/graphql) requires abusive struct tags for its reflection-based query generation, and the code generation tools produce mountains of boilerplate. Neither feels like writing Go. Both feel like fighting the language into doing something it actively resists.

<Conv name="Cadey" mood="coffee">
  GitHub removing the GraphQL explorer made this even worse. You used to be able
  to poke around the schema interactively and figure out what queries you
  needed. Now you're reading docs and guessing. Fun.
</Conv>

Every previous attempt at building this sponsor panel died in that swamp. I'd get partway through wrestling the GitHub Sponsors API into Go structs, lose momentum, and shelve it. Three separate attempts over the past year, all abandoned at roughly the same point: the moment where you realize the query you need is four levels of nested connections deep and the struct tags look like someone fell asleep on their keyboard.

So I tried vibe coding it as a hail mary. Worst case, it didn't work and I was no worse off than before. Best case, I'd ship something before disappearing into a hospital for a week.

## Preparing the skills

Vibe coding is not "type a prompt and pray." Output quality depends on the context you feed the model. Templ — the Go HTML templating library I use — barely exists in LLM training data. Ask Claude Code to write Templ components cold and it'll hallucinate syntax that looks plausible but doesn't compile. Ask me how I know.

<Conv name="Aoi" mood="wut">
  Wait, so how do you fix that?
</Conv>

Skills. I wrote four focused context documents that I could load into the agent's context window:

- **templ-syntax**: A quick overview of Templ's syntax with detailed resources so the model can look up specifics. Covers expressions, conditionals, loops, and how Go integration actually works.
- **templ-components**: How to break things into reusable components with props, children, and composition patterns. The stuff that's obvious if you've used Templ but impossible to infer from limited training data.
- **templ-htmx**: The weird edge cases when combining Templ with HTMX. There are gotchas around attribute rendering and event handling that trip up both humans and models.
- **templ-http**: Best practices for using Templ with `net/http` handlers. How to wire up routes, pass data to templates, and handle the request lifecycle properly.

With skills loaded, the model stops guessing at syntax and starts copying patterns from authoritative references. It doesn't need to _infer_ how to write Templ. It reads the skill, follows instructions, and produces code that compiles on the first try more often than not.

<Conv name="Mara" mood="hacker">
  Think of it like giving someone a cookbook instead of asking them to invent
  recipes from first principles. The ingredients are the same, but the results
  are dramatically more consistent.
</Conv>

## Building the thing

Next I pointed an agent team at a spec I'd written with [Mimi](https://mimi.xeserv.us). The spec covered the basics: OAuth login via GitHub, query the Sponsors API, render a panel showing who sponsors me and at what tier, store sponsor logos in [Tigris](https://www.tigrisdata.com/). Just the thing I'd failed to build by hand three times.

<Conv name="Cadey" mood="enby">
  I'm not going to pretend I wrote the spec alone. I talked through the
  requirements with Mimi and iterated on it until it was clear enough for an
  agent team to execute. The full spec is available as a gist if you want to
  see what "clear enough for agents" looks like in practice.
</Conv>

One agent team split the spec into tasks and started building. A second team reviewed the output and flagged issues. Meanwhile, I provisioned OAuth credentials in the GitHub developer settings, created the Neon Postgres database, and set up the Tigris bucket for sponsor logos. The workflow was async in the truest sense — agents would hit a point where they needed a credential, I'd paste it in, and they'd continue.

The GraphQL code the agents wrote is _ugly_. Raw query strings with manual JSON parsing that would make a linting tool weep. But it works. The shurcooL approach uses Go idioms, sure, but it requires so much gymnastics to handle nested connections that the cognitive load is worse. Agent-generated code is direct: send this query string, parse this JSON, done. I'd be embarrassed to show it at a code review. I'd also be embarrassed to admit how many times I failed to ship the "clean" version.

```go
// This is roughly what the agent generated.
```

## The stack

The full stack:

- **Go** for the backend, because that's what I know and what my site runs on
- **Templ** for HTML rendering, because I'm tired of `html/template`'s limitations
- **HTMX** for interactivity, because I refuse to write a React app for something this simple
- **PostgreSQL** via [Neon](https://neon.tech/) for persistence
- **GitHub OAuth** for authentication
- **GitHub Sponsors GraphQL API** for the actual sponsor data
- **[Tigris](https://www.tigrisdata.com/)** for sponsor logo storage — plugged it in and it Just Works™

## The warts

Org sponsorships are still broken. The schema for organization sponsors differs enough from individual sponsors that it needs its own query path and auth flow. I know what the fix looks like, but it requires reaching out to other devs who've navigated that particular maze.

The code also isn't my usual style. JSON parsing that makes me wince. Variable names that are functional but uninspired. Missing error context in a few places. I'll rewrite chunks of this after I've recovered. Even so, the panel _exists_ now. It renders real data. People can OAuth in and see their sponsorship status. Before this attempt, it was vaporware.

## What I actually learned

I wouldn't vibe code security-critical systems or anything I need to audit line-by-line. For breaking through a wall that had killed three previous attempts, though, it was exactly the right tool. Good-enough MVP, shipped before surgery. Done.

Skills make the difference. A model with skills follows instructions from authoritative references. A model without them guesses from training data, and the gap in output quality is enormous.

This sponsor panel probably won't look anything like it does today in six months. I suspect I'll rewrite the GraphQL layer once I find a pattern that doesn't make me cringe. Org sponsorships might work by then. HTMX might get replaced with something else entirely.

For now, the thing exists, and before my surgery, that mattered more than clean code.
