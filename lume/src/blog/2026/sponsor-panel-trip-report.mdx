A few weeks before I went under the knife, I had one of those moments where I looked at my project list and thought: if I don't ship this now, it's going to rot on the vine for months. I'm currently [on medical leave recovering from surgery](/blog/2026/killing-my-inner-necron/), and the window to get meaningful work done was closing fast.

The project in question: a sponsor panel at [sponsors.xeiaso.net](https://sponsors.xeiaso.net). Something that lets me see who's sponsoring me on GitHub, manage tiers, and generally have a dashboard for the financial side of my open source work. Not glamorous, but important enough that I'd been procrastinating on it for ages.

Here's the thing though -- I didn't build it the way I normally would. I vibe coded it. Directed an agent team, prepared skills, and let AI handle the parts that had defeated me multiple times before. It worked. I'm still kind of surprised.

## The GraphQL swamp

Go and GitHub's GraphQL API are oil and water. I've tried to build this sponsor panel before -- more than once -- and every attempt died the same death.

<Conv name="Cadey" mood="facepalm">
  Every. Single. Time.
</Conv>

Honestly, the GraphQL libraries in Go are genuinely awful. You're choosing between [shurcooL/graphql](https://github.com/shurcooL/graphql), which uses reflection-based struct tag gymnastics to build queries, or code generation tools that produce mountains of boilerplate you'll never fully understand. Neither option sparks joy. Both options make you question your career choices.

Making matters worse, GitHub removed their GraphQL explorer a while back, so you can't even interactively poke at the schema to figure out what fields exist. You're reading raw schema docs and guessing. It's the kind of developer experience that makes you close your laptop and go outside.

Previous attempts usually followed the same pattern: I'd get excited, spend an evening wrestling with struct tags, realize the pagination model for sponsors is more complex than I expected, get frustrated, and shelve it. Rinse and repeat every few months.

## The hail mary

With surgery looming, I figured: worst case, the vibe coding attempt fails and I'm exactly where I started. Best case, it actually works and I ship something before disappearing into a haze of anesthesia and hospital food.

Needless to say, it works. The panel is live. You can go look at it right now.

The stack ended up being Go, [Templ](https://templ.guide), [HTMX](https://htmx.org), and Postgres via [Neon](https://neon.tech). OAuth login through GitHub handles authentication. The GitHub Sponsors GraphQL API provides the actual sponsor data. Nothing exotic, but solid enough to ship on.

## Preparing the skills

Here's where this gets interesting. If you've been following my work with Claude Code, you know I've been experimenting with [agent skills](https://docs.anthropic.com/en/docs/claude-code/skills) -- essentially pre-loaded context documents that tell the model how to work with specific technologies.

[Templ](https://templ.guide) is not well-represented in LLM training data. It's a relatively young Go templating library and most models will confidently hallucinate syntax that looks plausible but doesn't compile. Without intervention, you get code that's 80% right and 100% broken.

Skills fix this. By loading reference documents into the context window, you give the model needles to copy from its haystack instead of forcing it to reconstruct syntax from vibes alone. The model doesn't need to _infer_ how Templ works -- it just follows what the skills say.

I made four skills for this project:

- **templ-syntax**: A quick overview of Templ's syntax with detailed resources for edge cases. This one was the most important -- it covers expressions, conditionals, loops, and how Go integration actually works versus how you'd expect it to work.
- **templ-components**: How to break UI elements into reusable components with props, children, and composition patterns. Without this, the model tends to dump everything into one giant template.
- **templ-htmx**: The weird edge cases when combining Templ with HTMX. There are subtle gotchas around how HTMX attributes interact with Templ's rendering model, and getting these wrong produces bugs that are maddening to debug.
- **templ-http**: Best practices for wiring Templ components into `net/http` handlers. This is the glue code that's easy to get subtly wrong in ways that only show up at runtime.

Loading these four skills transforms the model from "confidently wrong about Templ" to "reliably correct about Templ". It's the difference between fighting your tools and directing them.

## Specification

Before turning the agents loose, I worked with [Mimi](https://github.com/anthropics/claude-code) to draft a specification. This is something I want to emphasize: I didn't just say "build me a sponsor panel" and walk away. That's how you get a mess.

I described what I wanted -- the OAuth flow, the data I needed from the GraphQL API, the page structure, the tier management -- and we iterated on a spec together. Then I reviewed that spec and simplified it. Cut scope. Removed features I didn't actually need for an MVP.

The spec covered:

- GitHub OAuth login flow
- Querying the Sponsors GraphQL API for sponsor data (name, tier, avatar, URL)
- Storing sponsor data in Postgres for caching and display
- A simple dashboard showing current sponsors grouped by tier
- Session management with secure cookies

Nothing revolutionary. But having it written down meant the agent team had a clear target instead of vibing their way into feature creep.

## Implementation with an agent team

Once the spec was locked, I handed it to a second agent team. They split the spec into tasks and started building. Meanwhile, I did the one thing the agents couldn't do for themselves: I provisioned credentials.

GitHub OAuth client ID and secret. A Neon database URL. A personal access token with the `read:org` scope for the Sponsors API. The workflow went like this: agent hits an error because a credential is missing, I paste it in, agent continues. Rinse and repeat.

It felt like async pair programming where my partner was faster than me at writing code but couldn't open a browser to create an OAuth app. A fair tradeoff, honestly.

## UI v2: making it look like my website

The generic look lasted about an hour before I couldn't take it anymore. I did another pass with the agents to restyle everything to match my site's aesthetic -- the dark background, the fonts, the general vibe of xeiaso.net.

Most of it was a CSS and Templ component exercise. The agents handled it well because the skills I'd prepared covered exactly this kind of component-level UI work. Swapping out a bootstrap card for a custom styled component is the kind of task where Templ skills shine because the model knows the exact syntax for conditional classes, children composition, and attribute spreading.

## Org sponsorships

After the initial launch I did another pass to make organizational sponsorships work properly. This turned out to be gnarlier than individual sponsorships -- different schema shape, different fields available, different everything. GitHub treats org sponsors as a fundamentally different entity in the API, because of course they do.

Getting this right required another round of GraphQL wrestling, but at this point the pattern was established: let the agent generate the query, verify the data comes back correctly, move on. The code isn't pretty. There's some JSON parsing in there that would make a linting tool weep. But it handles both individual and organizational sponsors now, and that's what matters.

## The broader lesson

I'm genuinely happy with how this turned out. The sponsor panel exists, it works, and I shipped it before going into surgery. Mission accomplished.

Vibe coding isn't for everything. I wouldn't use this approach for something that needs to be deeply understood and maintained by a team. But for getting past a wall I'd banged my head against multiple times? For shipping a clean-enough MVP before disappearing for weeks of recovery? It's a real tool in the toolbox.

The skills are the missing piece that most people overlook when they talk about vibe coding. Without them, you're letting the model flail. With them, you're giving it a reference manual and saying "copy this, not whatever you hallucinated from your training data." That distinction is the difference between a frustrating afternoon and a working product.

I don't know if this sponsor panel will still look like this in six months. I suspect I'll rewrite the GraphQL code at minimum -- there's a version of this that uses proper query types instead of raw JSON parsing, and future-me will probably insist on it. But the panel exists, the data is correct, and my sponsors can see that I appreciate them.

Sometimes good enough really is good enough.
